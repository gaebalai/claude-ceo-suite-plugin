# 사용 가이드

Claude CEO-Suite Plugin을 일상 개발에서 활용하기 위한 실전 가이드입니다.

> 설치 방법은 [README.md](./README.md#설치)를 참조하세요.

---

## 60초 퀵스타트

1. 임의의 git 리포지토리에 `cd`
2. 다음을 실행:
   ```
   /claude-ceo-suite:ceo
   ```
3. 권장 사항을 확인. CEO 커맨드가 자동으로 프로젝트를 진단하고, 가장 관련성이 높은 3개의 경영 관점을 선택하여 우선순위가 매겨진 액션 리스트를 반환합니다.

그것으로 끝입니다. **어디서 시작할지 모르겠다면 `/ceo`부터 시작하세요** — 항상 올바르고 안전한 진입점으로 설계되어 있습니다.

## 3가지 진입점: `/ask`, `/<role>`, `/ceo`

「얼마나 스스로 생각할 것인가」의 수준에 따라 3가지 진입점이 있습니다:

| 진입점 | 관점 수 | 비용 | 사용 시기 |
|---|---|---|---|
| **`/claude-ceo-suite:ask <질문>`** | 1(자동 선정) | 최저 | 「어쨌든 답이 필요하다·역할 고르고 싶지 않다」 |
| **`/claude-ceo-suite:<role> <질문>`** | 1(직접 지정) | 저렴 | 「이건 CTO 안건이라는 걸 알고 있다」 |
| **`/claude-ceo-suite:ceo <질문>`** | 3 + 통합 | 최고 | 「복수 관점의 트레이드오프 해결이 필요하다」 |

`/ask`는 최신이자 가장 심플한 방법. 질문을 11개 역할 전체에 대해 스코어링하고, 가장 적합한 1명을 선택하여 그 관점으로 답변합니다. 질문이 정말로 복수 관점을 필요로 하는 경우(예: 「지금 릴리스해야 하나 먼저 강화해야 하나」), `/ask`는 **라우팅을 거부**하고 `/ceo`를 안내합니다. 이 정직한 거부는 기능의 일부입니다 — 부적절한 단일 관점의 답변은 리다이렉트보다 해롭기 때문입니다.

```
/claude-ceo-suite:ask 이 SQL 스키마는 정규화되어 있나요?
/claude-ceo-suite:ask 새로 추가한 의존성은 얼마나 위험한가요?
/claude-ceo-suite:ask README 구조가 SEO적으로 좋은가요?
```

## 2가지 모드: 질문과 리뷰

모든 커맨드는 2가지 모드로 동작합니다. 플러그인이 입력으로부터 어느 쪽을 사용할지 자동 감지합니다.

### 질문 모드

자연어로 질문하기만 하면 됩니다. 실제 코드베이스에 기반하여 해당 직책의 관점에서 답변합니다.

```
/claude-ceo-suite:cto 모노레포로 전환해야 할까요?
/claude-ceo-suite:cso JWT 구현이 안전한가요?
/claude-ceo-suite:clo 이 GPL 라이브러리를 SaaS에서 사용할 수 있나요?
/claude-ceo-suite:cfo 데이터베이스 티어가 오버프로비저닝되어 있나요?
/claude-ceo-suite:pm 이 버그를 고치기 위해 릴리스를 연기해야 할까요?
```

직책은 질문에 답하기 위해 필요한 컨텍스트만 수집합니다 — 빠르고 초점이 맞춰져 있습니다.

### 리뷰 모드

스코프 키워드(또는 아무것도 없이)를 전달하면 구조화된 분석이 시작됩니다.

```
/claude-ceo-suite:cto                 # 전체 리뷰
/claude-ceo-suite:cto debt            # 기술 부채에 집중
/claude-ceo-suite:cso auth            # 인증에 집중
/claude-ceo-suite:clo licenses        # 의존성 라이선스에 집중
/claude-ceo-suite:coo cicd            # CI/CD 파이프라인에 집중
/claude-ceo-suite:cmo seo             # SEO에 집중
/claude-ceo-suite:caio models         # 모델 라이프사이클에 집중
/claude-ceo-suite:cfo costs           # 클라우드 비용에 집중
/claude-ceo-suite:cio data            # 데이터 거버넌스에 집중
```

각 커맨드가 받아들이는 스코프는 커맨드 파일의 프론트매터에 기재되어 있습니다(인수 없이 실행하면 기본 리뷰를 볼 수 있습니다).

## 자주 쓰는 패턴

### 복수 역할로 상담

같은 세션에서 여러 역할을 실행하면, 각 역할은 이미 호출한 다른 역할의 분석 결과를 자동 참조합니다.

**예: 릴리스 전 준비 흐름**

```
1. /claude-ceo-suite:cto       → 기술 부채의 블로커를 식별
2. /claude-ceo-suite:cso       → 부채 수정이 인증 갭도 해소한다는 것을 발견
3. /claude-ceo-suite:qa-lead   → 인증 수정 주변에서 미테스트 코드 경로를 발견
4. /claude-ceo-suite:pm        → 위의 모든 사항을 반영하여 마일스톤을 재정리
```

`/pm`을 실행하는 시점에서 `cto`, `cso`, `qa-lead`의 소견이 컨텍스트에 있으며, 이를 우선순위 매기기에 반영합니다.

### CEO에게 사령탑을 맡기기

어떤 역할에 상담해야 할지 모르겠다면, 먼저 `/ceo`를 실행하세요. 상황에 가장 적합한 3개의 관점을 선택하고, 통합된 의사결정으로 반환합니다. 그 후 필요에 따라 개별 역할로 파고들 수 있습니다.

```
/claude-ceo-suite:ceo 런칭 준비가 되어 있나요?
```

출력에는 선택된 3명의 CxO와 그 이유가 표시되므로, 다음에 무엇을 호출해야 할지의 좋은 가이드가 됩니다.

### 스코프를 좁혀 시간 단축

큰 리포지토리에서는 전체 리뷰에 시간이 걸립니다. 스코프 키워드로 좁히세요.

```
/claude-ceo-suite:cso auth        # 인증 코드만, 리포 전체가 아닌
/claude-ceo-suite:cto debt        # 기술 부채만, 문서/버그/아키텍처는 스킵
```

## 이 플러그인이 하지 않는 것

- **코드 변경은 하지 않음** — 모든 커맨드는 분석과 권장만 반환합니다. 무엇을 실행할지는 여러분이 결정합니다.
- **권위 있는 법무/보안/재무 조언이 아님** — 출력은 AI 생성입니다. 중요한 권장 사항은 행동 전에 전문가에게 확인하세요. 모든 커맨드 출력의 마지막에 면책 조항이 있습니다.
- **텔레메트리 없음** — 플러그인은 외부로 데이터를 전송하지 않습니다. 여러분의 기존 인증 정보로 `gh`와 `git` 명령을 호출할 뿐입니다.

## 트러블슈팅

### 「질문했는데 리뷰 모드로 답변받았다」(또는 그 반대)

플러그인은 입력에 의문사나 `?`가 포함되어 있는지로 질문/리뷰를 자동 감지합니다. 감지가 잘못된 경우:

- **강제로 리뷰 모드로**: 스코프 키워드를 전달
  ```
  /claude-ceo-suite:cto full
  ```
- **강제로 질문 모드로**: 입력 끝에 `?`를 붙임
  ```
  /claude-ceo-suite:cto main 브랜치를 다시 리뷰할까요?
  ```

### 「`--repo owner/repo`를 사용하려다 실패했다」

오래된 버전을 사용하고 있습니다. 플러그인은 `owner/repo` 인수를 더 이상 받지 않습니다 — 현재 워킹 디렉토리의 리포지토리에서 동작합니다. 대상 리포지토리로 `cd`하거나 v1.4 이상으로 업그레이드하세요.

### 「존재하지 않는 파일을 참조하고 있다」

LLM의 할루시네이션은 특히 대규모 코드베이스에서 발생할 수 있습니다. 행동 전에 반드시 참조를 검증하세요. 할루시네이션이 자주 발생하면 커맨드 이름·입력·실행한 모델을 첨부하여 Issue를 보고하세요.

### 「13개 역할 외에 커스텀 역할이 필요하다」

## 메인테이너용: audit 커맨드

이 플러그인(또는 그 포크)을 유지보수하는 경우, audit 커맨드를 릴리스 게이트로 사용하세요:

```
/claude-ceo-suite:audit              # 감사를 실행하고 합격/실패를 보고
/claude-ceo-suite:audit matrix       # 정합성 매트릭스도 함께 표시
```

감사에 합격하면 릴리스 절차가 안내됩니다. 실패한 경우 각 실패에 대해 수정안이 제시되며, 수정 후 다시 실행하도록 안내됩니다. audit 커맨드는 읽기 전용으로, 파일을 일절 변경하지 않습니다.

터미널에서 직접 스크립트를 실행할 수도 있습니다:

```bash
python3 scripts/audit.py
python3 scripts/audit.py --matrix
```

모든 체크 목록은 [AUDIT.md](./AUDIT.md)를 참조하세요.
